DevBuddy Usage Guide

คู่มือใช้งาน DevBuddy สำหรับ workflow แบบ senior developer, agent orchestration, skill routing, และการประหยัด token ใน Codex

ภาพรวม

DevBuddy คือ skill/workflow ที่ทำให้ AI coding assistant ทำงานเหมือน senior developer: เข้าใจ business ก่อน, อ่าน architecture ก่อน, วางแผนก่อนแก้, ทำ change ให้น้อยที่สุด, review งาน, และบันทึกความรู้ที่ควรจำไว้ใน memory files

หลักสำคัญคือ main agent เป็น orchestrator ส่วน agent ย่อยใช้เฉพาะงานที่ช่วยลดความเสี่ยงหรือลด context load ได้จริง

Workflow หลัก

  1. รับคำสั่งจากผู้ใช้
  2. อ่าน SKILL.md
  3. อ่าน memory ตามลำดับ: Context.md, BusinessContext.md, DecisionLog.md, KnowledgeBase.md
  4. อ่าน ADR หรือเอกสารที่เกี่ยวข้องถ้ามี
  5. สำรวจ source code และ tests เฉพาะส่วนที่เกี่ยวข้อง
  6. ถามเฉพาะคำถามที่มีผลต่อความถูกต้องจริง ๆ
  7. วางแผนก่อน implementation เมื่อเป็นงานแก้โค้ด
  8. ทำ change ที่เล็กและ maintainable ที่สุด
  9. review correctness, business alignment, simplicity, security, performance, และ testability
  10. อัปเดต memory เมื่อมี durable knowledge ใหม่

บทบาท Agent

Role หน้าที่ ผลลัพธ์ที่ควรส่งกลับ
Main Agent วิเคราะห์คำสั่ง, อ่าน context หลัก, เลือก skill, route งาน, รวมผล, ตัดสินใจ, สื่อสารกับผู้ใช้ แผนหรือคำตอบสุดท้ายที่รวม business, technical, risks, tests, และ next steps
Analyze Agent สำรวจ repo/source/tests แบบ read-only, หา affected area, risks, unknowns summary, inspected files, findings, risks, confidence, recommended next steps
Frontend Agent งาน Angular/UI, accessibility, responsive behavior, routing, state, forms, frontend tests UI behavior, affected files, accessibility/visual risks, verification notes
Backend Agent งาน .NET/API/data/persistence/security/runtime/performance/backend tests API/runtime behavior, affected files, data/security/performance risks, verification notes

การเรียก Agent ย่อย

ควรเรียกเมื่อ

  • งานต้องสำรวจหลาย module หรือหลาย concern
  • ต้องการ review ความเสี่ยงหรือ regression แยกอีกมุม
  • งาน frontend/backend มีรายละเอียดเฉพาะทางชัดเจน
  • การแบ่งงานช่วยลด context ที่ main agent ต้องโหลดเอง

ไม่ควรเรียกเมื่อ

  • งานเล็กและ main agent มี context พอแล้ว
  • ต้องส่ง context กว้างมากจน overhead สูงกว่าประโยชน์
  • agent หลายตัวจะต้องอ่านไฟล์ชุดเดียวกันซ้ำ ๆ

Context Contract

เมื่อส่งงานให้ agent ย่อย ให้ส่งเฉพาะข้อมูลที่จำเป็น:

  • เป้าหมายและ success criteria
  • ไฟล์ โฟลเดอร์ symbol หรือ workflow ที่เกี่ยวข้อง
  • ข้อจำกัดว่าแก้ไฟล์ได้หรือไม่
  • รูปแบบ output ที่ต้องการ
  • business rules, assumptions, unknowns ที่รู้แล้ว
หลีกเลี่ยงการส่งทั้ง repo, log ยาว ๆ, หรือ memory ที่ไม่เกี่ยวข้องให้ agent ย่อย เพราะจะเปลือง token และเพิ่ม noise

Memory Files

  • Context.md: architecture, modules, data flow, dependencies, runtime behavior, test strategy
  • BusinessContext.md: domains, users, workflows, business rules, edge cases, compliance constraints
  • DecisionLog.md: decisions, alternatives, trade-offs, rationale
  • KnowledgeBase.md: lessons learned, anti-patterns, bug prevention, proven solutions

อัปเดต memory เฉพาะเมื่อมีความรู้ที่ควรจำข้ามงาน และต้อง preserve ของเดิม ไม่ overwrite ทั้งไฟล์โดยไม่จำเป็น

Codex และ RTK

ใน Codex adapter ให้แยกกติกาเฉพาะแพลตฟอร์มไว้ใน agents/codex/operations.md เพื่อไม่ให้ shared policy ปนกับ behavior เฉพาะ Codex

Plan Mode

  • ใน Plan Mode ให้ถือว่า repo-tracked files เป็น read-only
  • ใช้ agent ย่อยเพื่อ discovery, validation, risk analysis เท่านั้น
  • ต้องออกแผน approval-ready ก่อน implementation

RTK บน PowerShell

ใช้ rtk เพื่อลด output จาก command ตาม workspace convention

rtk rg --files
rtk git status --short
rtk proxy powershell -NoProfile -Command "Get-Content -LiteralPath 'C:\path\file.md'"

ตัวอย่าง Prompt

งานวิเคราะห์ก่อนแก้

ใช้ DevBuddy วิเคราะห์ feature login ก่อนแก้โค้ด
ขอ affected files, risks, missing tests, และ implementation plan

งาน frontend

ปรับหน้า dashboard ให้ responsive ดีขึ้น
ใช้ frontend profile ตรวจ existing UI pattern ก่อน แล้วค่อยเสนอ plan

งาน backend

เพิ่ม validation ให้ API create order
ตรวจ business rules, API contract, persistence impact, และ tests ที่ควรเพิ่ม

งาน review

review change นี้แบบ DevBuddy
ให้ findings ก่อน summary และระบุ file/line references ถ้ามี

Checklist ก่อนจบงาน

  • เข้าใจ business goal และ affected workflow แล้ว
  • อ่าน source/tests เฉพาะส่วนที่เกี่ยวข้องแล้ว
  • มี plan หรือ rationale ก่อนแก้โค้ด
  • ใช้ subagent เฉพาะเมื่อคุ้มกับ token overhead
  • output ของ subagent กระชับ มี file refs, risks, confidence
  • change มีขอบเขตเล็กและ maintainable
  • tests หรือ validation เหมาะกับความเสี่ยง
  • memory files อัปเดตเมื่อมี durable knowledge
