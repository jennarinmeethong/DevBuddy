DevBuddy Skill Reference

DevBuddy คือ skill ที่เปลี่ยน AI coding assistant ให้ทำงานแบบ senior developer: เข้าใจ business และ architecture ก่อน, วางแผนก่อนแก้, เปลี่ยนให้น้อยแต่ดี, review งานจริงจัง, และเก็บความรู้ที่ควรจำไว้ใช้ต่อในงานถัดไป หน้านี้เป็น wiki/reference สำหรับดูว่า skill ทำอะไร, ใช้งานอย่างไร, route sub agent อย่างไร, และต้องดูแลไฟล์ไหนเมื่อเนื้อหา skill เปลี่ยน

สารบัญ

DevBuddy คืออะไร

DevBuddy เป็น workflow skill สำหรับ AI coding assistant ที่เน้นการตัดสินใจเชิงวิศวกรรม ไม่ใช่แค่สั่งให้ AI เขียนโค้ดเร็วขึ้น แต่บังคับให้ AI เข้าใจปัญหา บริบท ความเสี่ยง และวิธี verify ก่อนลงมือ

แก่นของ skill นี้คือ main agent เป็น orchestrator: อ่านโจทย์, เลือกข้อมูลที่ต้องใช้, route งานให้ sub agent เมื่อเหมาะสม, รวมผล, ตัดสินใจ, และสื่อสารกับผู้ใช้

Business before code

ก่อนแก้โค้ด DevBuddy ต้องเข้าใจ goal, target user, workflow ที่กระทบ, domain rules, edge cases, และ expected behavior

BA and SA ready

Skill ปัจจุบันแยก business analysis และ system analysis เป็น role ชัดเจน: ba จับ requirement และ value, sa จับ architecture และ contracts

Progressive context

โหลดเฉพาะข้อมูลที่จำเป็น: เริ่มจาก SKILL.md, memory เมื่อเกี่ยวข้อง, แล้วค่อยอ่าน source, tests, ADR หรือ references

Small maintainable changes

ลำดับคิดหลักคือ Delete, Simplify, Reuse, Refactor, Build และเพิ่ม abstraction เฉพาะเมื่อมีเหตุผลจริง

ข้อมูลจำเป็นก่อนใช้

ถ้าเพิ่งเปิด repo นี้เป็นครั้งแรก ให้เริ่มจากส่วนนี้ก่อน เพราะเป็นจุดที่ทำให้เข้าใจว่า DevBuddy ถูกโหลดจากไหน ใช้เมื่อไหร่ และควรคาดหวัง output แบบใดจาก agent

คิดว่า DevBuddy เป็น workflow layer ที่ครอบการทำงานของ coding agent ไม่ใช่ framework, CLI, package manager หรือ test runner ใหม่

Skill entrypoint

ไฟล์หลักคือ skill/SKILL.md ใช้บอก agent ว่าต้องคิดแบบ business-first, plan ก่อน implementation, review หลังแก้ และ update memory เมื่อมีความรู้ที่ควรจำ

Claude Code loading path

คำสั่ง /devbuddy จะเห็นได้เมื่อมี skill ที่ .claude/skills/devbuddy/SKILL.md ใน project หรือ ~/.claude/skills/devbuddy/SKILL.md สำหรับ global install

Package path

packages/claude/devbuddy/ เป็น layout สำหรับแจกจ่ายหรือสร้าง zip ไม่ใช่ path ที่ Claude Code auto-load จาก repo โดยตรง

Agent model

Main agent เป็นคนคุมภาพรวม ส่วน sub agent เป็น role เฉพาะทางที่ถูกเรียกเมื่อช่วยลด risk, ลด context load หรือช่วยตรวจคนละมุม

Memory model

Memory files ไม่ได้ถูกอ่านทุกงาน จะโหลดเฉพาะเมื่อมีผลต่อ project-specific context, business rules, prior decisions หรือ lessons learned

Expected behavior

งาน implementation ควรมี plan, risks, alternatives, confidence และ verification ก่อนหรือระหว่างลงมือ ไม่ใช่แก้โค้ดทันทีโดยไม่อ่านบริบท

Command and test rules

ถ้า command ที่ตั้งใจใช้ไม่มีในเครื่อง agent ต้องถามก่อนเปลี่ยน tool หรือ package manager และถ้าต้องสร้าง test ใหม่ ให้ยึด layout เดิม หรือใช้ tests/ ระดับบนเมื่อไม่มี convention ชัดเจน

คำสั้น ๆ ที่ใช้เรียก skill

สถานการณ์ ตัวอย่างคำขอ
อยากให้ agent ทำงานตาม workflow หลัก /devbuddy ช่วยวิเคราะห์และวางแผนก่อนแก้ feature นี้
อยากได้ review แบบเข้ม review PR นี้แบบ DevBuddy ให้ findings ก่อน summary
อยากใช้ role เฉพาะทาง ใช้ DevBuddy frontend profile ตรวจ responsive และ accessibility risk
อยาก refine requirement ใช้ DevBuddy ba profile หา stakeholders, workflow, domain rules และ acceptance criteria
อยากออกแบบ system behavior ใช้ DevBuddy sa profile map modules, APIs, data flow, risks และ alternatives
อยากออกแบบหรือรัน test ใช้ DevBuddy tester profile ออก test cases, test data และ evidence summary
อยากเข้าใจ codebase ก่อนทำ ใช้ DevBuddy analyze profile map affected files, risks และ missing tests

ใช้ทำอะไรได้บ้าง

ใช้ DevBuddy เมื่อต้องการให้ AI ทำงานกับ codebase อย่างระมัดระวัง มีเหตุผล และทิ้งความรู้ที่ใช้ซ้ำได้ ไม่ใช่แก้เฉพาะหน้าแล้วหายไป

เหมาะกับงานที่ต้องมี BA/SA thinking, planning, review, testing discipline, Angular/.NET guidance, หรือการประสาน sub agent หลายบทบาท

งาน business analysis

  • สรุป business problem, target users, stakeholders และ intended value
  • ทำ requirement discovery, acceptance criteria, out-of-scope และ material questions
  • ระบุ domain rules, workflow impact, priority และ readiness ก่อน implementation

งาน system analysis

  • แปลง requirement เป็น system behavior, module boundaries และ API/integration contracts
  • ดู data flow, runtime behavior, non-functional requirements และ architecture risks
  • เสนอ alternatives, compatibility notes, validation points หรือ ADR/memory updates

งานวิเคราะห์และวางแผน

  • อ่าน feature หรือ module เพื่อหา current behavior
  • ระบุ affected files, risks, unknowns, และ missing tests
  • ทำ implementation plan ก่อนแก้จริง

งาน implementation

  • แก้โค้ดแบบ scope เล็กและ maintainable
  • คุม business rules, architecture, security, performance และ testability
  • เลือก Angular หรือ .NET reference เฉพาะเรื่องที่เกี่ยวข้อง

งาน review, QA และ testing

  • ตรวจ bug, regression, missing tests และ acceptance gaps
  • ให้ findings ก่อน summary เมื่อเป็น code review หรือ QA review
  • ออกแบบ test cases, test data, defect reproduction steps และ evidence summary

งาน documentation และ learning

  • อัปเดต README, usage guide, migration notes, skill docs และ adapter docs
  • บันทึก durable knowledge ใน memory files เมื่อควรจำข้ามงาน
  • รักษาความสอดคล้องระหว่าง shared skill และ Claude Code adapters
DevBuddy ไม่ได้ทำให้ zip package ถูก auto-load จาก repo เอง ถ้าจะใช้ใน Claude Code ต้องมี skill อยู่ที่ <project>/.claude/skills/devbuddy/SKILL.md หรือ ~/.claude/skills/devbuddy/SKILL.md

DevBuddy ทำงานอย่างไร

Skill นี้ออกแบบแบบ progressive disclosure: ให้ AI เริ่มจาก instruction หลักก่อน แล้วค่อยดึง context เพิ่มเฉพาะเมื่อจำเป็น เพื่อประหยัด token และลด noise

สำหรับคำถามทั่วไปเกี่ยวกับ skill ให้ inspect เฉพาะ instruction หรือ resource ที่เกี่ยวข้อง ไม่ต้องอ่าน project memory, ADR, source code หรือ tests โดยไม่จำเป็น
1

Classify

อ่านคำขอและแยกประเภทงาน: question, analysis, planning, implementation, review, debugging หรือ learning

2

Load Context

อ่าน SKILL.md แล้วโหลด memory, ADR, source, tests หรือ references เฉพาะที่มีผลต่อความถูกต้อง

3

Plan

สรุป business goal, technical approach, affected files, risks, alternatives, questions และ confidence

4

Implement

ทำ change ที่เล็กที่สุดซึ่งยัง maintainable และสอดคล้องกับ architecture เดิม

5

Review

ตรวจ correctness, business alignment, simplicity, security, performance, testability และ regression risk

6

Verify

รัน test หรือ validation ที่เหมาะกับ blast radius และอธิบาย test ที่ไม่ได้รัน

7

Learn

ถ้ามี durable knowledge ให้บันทึกลง memory โดย preserve ความรู้เดิมและ mark assumption ให้ชัด

8

Report

รวมผลจาก main agent และ sub agent เป็นคำตอบสุดท้ายที่ actionable และมี risks/tests ชัดเจน

Planning output ที่ DevBuddy คาดหวัง

หัวข้อ ต้องตอบอะไร
Business summary ปัญหาของผู้ใช้หรือธุรกิจ และ workflow ที่ได้รับผลกระทบ
Technical summary แนวทาง implementation และ expected behavior หลังแก้
Affected files ไฟล์ โฟลเดอร์ module หรือพื้นที่ที่คาดว่าจะเปลี่ยนหรืออ่าน
Risks LOW, MEDIUM, HIGH หรือ CRITICAL พร้อมเหตุผลสั้น ๆ
Alternatives อย่างน้อยแนวทางที่เลือกและทางเลือกที่ไม่เลือก
Open questions เฉพาะคำถามที่กระทบ correctness จริง
Confidence LOW, MEDIUM หรือ HIGH พร้อมเหตุผล

Sub Agent คืออะไรใน DevBuddy

Sub agent คือ role เฉพาะทางที่ main agent เรียกใช้เมื่อการแบ่งงานช่วยลดความเสี่ยง ลด context load หรือให้มุมมองเฉพาะด้านที่ดีกว่า แต่ main agent ยังต้องรับผิดชอบภาพรวม การตัดสินใจ และคำตอบสุดท้าย

อย่าเรียก sub agent เพียงเพราะมีให้ใช้ งานเล็ก ๆ ที่ main agent มี context พอแล้ว ควรทำตรง ๆ เพื่อเลี่ยง overhead และการอ่านไฟล์ซ้ำ

Main Agent รับผิดชอบ

  • อ่านคำขอและ classify งาน
  • ตัดสินใจว่าจะ route งานหรือทำเอง
  • ส่ง context ให้ sub agent เฉพาะเท่าที่จำเป็น
  • รวม findings เป็น plan หรือ final answer เดียว
  • รักษา planning, approval, review, memory และ definition-of-done rules

Sub Agent รับผิดชอบ

  • ทำงานเฉพาะบทบาท เช่น analyze, BA, SA, frontend, backend, QA หรือ tester
  • อ่านไฟล์/คำสั่งเท่าที่ task-local context ระบุ
  • คืนผลแบบ concise พร้อม file refs, risks, unknowns และ confidence
  • เสนอ memory update ให้ parent workflow เมื่อควรจำ
  • ไม่แทนที่ final judgment ของ main agent

Sub agent lifecycle

1

Decide

Main agent ประเมินว่างานนี้ควรทำเองหรือ route ให้ role เฉพาะทาง เพราะมี risk, scope หรือ context กว้าง

2

Brief

ส่ง goal, success criteria, relevant paths, constraints และ output shape ให้แคบที่สุดที่ยังทำงานได้

3

Inspect

Sub agent อ่านเฉพาะไฟล์หรือคำสั่งที่เกี่ยวข้อง แล้วคืน findings, risks, unknowns และ confidence

4

Integrate

Main agent รวมผลกับ business context, ตัดสินใจแนวทาง, ทำ plan/final answer และรับผิดชอบคำตอบสุดท้าย

เมื่อไหร่ควรเรียก sub agent

ควรเรียก

  • ต้องสำรวจหลาย module หรือหลาย concern
  • ต้องการ read-only analysis ก่อนแก้จริง
  • งาน BA/SA/frontend/backend/data/tester มีรายละเอียดเฉพาะทางชัดเจน
  • ต้องการ QA pass เพื่อหา regression หรือ missing tests หรือ tester pass เพื่อออกแบบ/รัน test cases
  • การแบ่งงานช่วยให้ main agent ไม่ต้องโหลด context กว้างเกินไป

ไม่ควรเรียก

  • งานเล็กและมี context ครบแล้ว
  • ต้องส่ง repo context กว้างมากจน overhead สูง
  • sub agent หลายตัวจะต้องอ่านไฟล์ชุดเดียวกันซ้ำ
  • ต้องการ operations โดยผู้ใช้ไม่ได้ระบุ DevOps/CI/CD/deployment/runtime intent
  • ผลลัพธ์ที่ต้องการเป็นคำตอบสั้น ๆ ไม่ใช่งานวิเคราะห์หรือเปลี่ยนโค้ด

Sub agent ไม่ใช่อะไร

ไม่ใช่คนตัดสินใจแทน

Sub agent ให้ evidence และ recommendation ได้ แต่ main agent ยังต้องเป็นคนชั่ง trade-off, ผูกกับ business goal และสื่อสาร final answer

ไม่ใช่ข้ออ้างให้อ่านทุกอย่าง

การ route งานต้องลด context load ไม่ใช่เพิ่ม noise ถ้าต้องส่ง repo ทั้งหมดให้ sub agent อาจแปลว่างานยังไม่ถูกแบ่งขอบเขต

ไม่ใช่ role operations อัตโนมัติ

operations ใช้เฉพาะเมื่อผู้ใช้ระบุ DevOps, CI/CD, Docker, deployment, hosting, release หรือ runtime environment อย่างชัดเจน

Mutation constraints บอกชัดว่า read-only หรืออนุญาตให้แก้ไฟล์บางส่วน
Output shape ทำให้ผลลัพธ์รวมกลับเข้า main workflow ได้ง่าย
Known business rules, assumptions, unknowns ป้องกันการเดา business rule ใหม่หรือวิเคราะห์ซ้ำโดยไม่จำเป็น

Output contract ของ sub agent

  • สรุป business และ technical impact แบบกระชับ
  • ระบุ files หรือ areas ที่ inspect, change หรือ recommend
  • ให้ findings พร้อม file references เมื่อมีหลักฐานในไฟล์
  • ระบุ risks, unknowns, confidence และ recommended next steps/tests
  • ไม่ paste full files หรือ logs ยาว ๆ ให้ summarize และ cite จุดที่สำคัญที่สุด

Memory files

DevBuddy มี memory files สำหรับเก็บความรู้ที่ควรอยู่ข้ามงาน แต่ไม่ได้อ่านทุกครั้ง จะอ่านเฉพาะเมื่อช่วยเรื่อง project-specific context, business rules, prior decisions หรือ lessons learned

ลำดับอ่านเมื่อจำเป็นคือ Context.md, BusinessContext.md, DecisionLog.md, แล้ว KnowledgeBase.md
ไฟล์ ใช้เก็บอะไร ควรอัปเดตเมื่อ
skill/Context.md architecture, modules, data flow, dependencies, runtime behavior และ test strategy มี technical understanding ใหม่ที่ช่วยงานถัดไป
skill/BusinessContext.md domains, users, workflows, business rules, edge cases และ compliance constraints เจอ business rule หรือ domain behavior ที่ควรจำถาวร
skill/DecisionLog.md durable decisions, trade-offs, alternatives และ rationale มี architectural หรือ product/technical decision ที่ควรอ้างอิงภายหลัง
skill/KnowledgeBase.md lessons learned, anti-patterns, bug prevention และ proven solutions พบ pattern ที่ช่วยป้องกัน bug หรือทำงานเร็วขึ้นในอนาคต
อัปเดต memory เฉพาะข้อมูลที่ durable จริง และต้อง preserve ของเดิมเสมอ ถ้าเป็น assumption ให้ระบุว่าเป็น assumption ไม่ใช่ข้อเท็จจริง

วิธีใช้งาน

ถ้าใช้ใน Claude Code และ path พร้อมแล้ว ให้เรียก /devbuddy หรือบอกให้ AI ใช้ DevBuddy workflow ในโจทย์โดยตรง

ถ้าเพิ่ม .claude/skills/ หลังเปิด Claude Code แล้ว ให้ restart Claude Code จาก repo root เพื่อให้เห็น /devbuddy

Activation checklist

ต้องการใช้งานแบบไหน ต้องมีอะไร เช็คอย่างไร
ใช้ใน repo นี้ .claude/skills/devbuddy/SKILL.md เปิด Claude Code จาก repo root แล้วเรียก /devbuddy
ใช้เป็น global skill ~/.claude/skills/devbuddy/SKILL.md restart Claude Code หลังติดตั้ง แล้วตรวจว่า slash command ปรากฏ
แจกจ่ายให้ project อื่น package หรือ zip ที่ rebuild จาก source ล่าสุด รัน prepare-zips จาก repository root แล้วติดตั้ง path ปลายทางให้ถูก
ใช้ sub agent adapters .claude/agents/devbuddy-*.md และ skill/agents/shared/ ตรวจชื่อ role, scope, constraints และ output ให้ตรงกับ shared spec

Prompt examples

งาน BA

ใช้ DevBuddy ba profile refine requirement นี้
สรุป target users, affected workflow, domain rules,
acceptance criteria, assumptions และ open questions

งาน SA

ใช้ DevBuddy sa profile วิเคราะห์ architecture impact
map module boundaries, API contracts, data flow,
non-functional risks, alternatives และ validation points

วิเคราะห์ก่อนแก้

ใช้ DevBuddy วิเคราะห์ feature login ก่อนแก้โค้ด
ขอ current behavior, affected files, risks, missing tests,
implementation plan และ confidence

งาน frontend

ใช้ DevBuddy ปรับหน้า dashboard ให้ responsive ดีขึ้น
ให้ frontend profile ตรวจ existing UI pattern,
accessibility risk และ test ที่ควร verify ก่อนเสนอ plan

งาน backend

ใช้ DevBuddy เพิ่ม validation ให้ API create order
ตรวจ business rules, API contract, persistence impact,
security concern และ tests ที่ควรเพิ่ม

งาน review

review change นี้แบบ DevBuddy
ให้ findings ก่อน summary, ระบุ file/line references,
test gaps, regression risks และ recommended verification

งาน tester

ใช้ DevBuddy tester profile ออก test cases สำหรับ checkout flow
ระบุ preconditions, test data, expected results,
regression cases และ evidence summary ที่ควรเก็บ

งาน documentation

ใช้ DevBuddy docs profile ทำ usage guide ของ skill นี้
ให้ตรวจ SKILL.md, shared agent specs, package path
และสรุป maintenance notes ที่คนใหม่ต้องรู้

งาน data

ใช้ DevBuddy data profile วิเคราะห์ schema change นี้
ระบุ data ownership, compatibility risk,
migration impact และ rollback concern

งาน operations แบบ explicit

ใช้ DevBuddy operations profile ตรวจ CI/CD pipeline นี้
เพราะโจทย์นี้เกี่ยวกับ deployment readiness,
runtime environment และ release risk โดยตรง

Maintenance notes

DevBuddy ถูกดูแลเป็น skill เดียวที่มีหลาย adapter ดังนั้นการแก้ behavior ต้องเริ่มจาก shared source แล้วค่อยตรวจ alignment ของ Claude Code path และ package copy

ถ้าแก้ role behavior ให้แก้ skill/agents/shared/ ก่อน แล้วตรวจ .claude/agents/ ที่ตรงกัน
ต้องการเปลี่ยน แก้ที่ไหนก่อน ต้องตรวจอะไรต่อ
Workflow หลัก, planning, memory policy, definition of done skill/SKILL.md .claude/skills/devbuddy/SKILL.md และ package copy หลัง rebuild
Scope, constraints หรือ output ของ sub agent skill/agents/shared/*.md adapter ที่ตรงกันใน .claude/agents/
Claude Code-specific prompt, tools หรือ frontmatter .claude/agents/*.md shared spec ต้องยังไม่ขัดกับ adapter
Distribution package root source files แล้ว run prepare-zips packages/claude/devbuddy/ และ zip archive ที่ build ใหม่
อย่าแก้เฉพาะไฟล์ใน packages/claude/devbuddy/ แล้วลืม source หลัก เพราะ package layout เป็นผลลัพธ์สำหรับ install/distribution และควร rebuild ได้

ก่อนปล่อยหรือส่งต่อให้คนอื่น

  • ตรวจว่า skill/SKILL.md และ .claude/skills/devbuddy/SKILL.md ยังสื่อ behavior เดียวกัน
  • ตรวจว่า role ใน skill/agents/shared/ ตรงกับ adapter ใน .claude/agents/
  • ตรวจว่า manual นี้ยังอธิบาย path, command, role และ package layout ตรงกับ repo ปัจจุบัน
  • rebuild package layout หรือ zip เมื่อมีการเปลี่ยน source ที่ต้องแจกจ่าย
  • เปิดไฟล์ HTML ใน browser หลังแก้ เพื่อดูว่าตาราง โค้ด และข้อความยาวยังอ่านได้บนจอเล็ก

Checklist สำหรับคนใช้ DevBuddy

ใช้รายการนี้เช็คว่า agent ทำงานตามเจตนาของ skill แล้วหรือยัง

Definition of done ของ DevBuddy ต้องครอบคลุม business requirements, technical requirements, verification, risks, review และ memory เมื่อเกี่ยวข้อง
  • อ่าน SKILL.md แล้วก่อนเริ่มงาน
  • classify งานถูกประเภทแล้ว
  • เข้าใจ business goal และ affected workflow แล้ว
  • โหลด memory เฉพาะเมื่อมีประโยชน์จริง
  • อ่าน source/tests เฉพาะพื้นที่ที่เกี่ยวข้อง
  • ระบุ unknowns และถามเฉพาะคำถามที่มีผลต่อ correctness
  • มี plan ก่อน implementation เมื่อเป็นงานแก้โค้ด
  • เลือก sub agent เมื่อช่วยลด risk หรือ context load จริง
  • ส่ง context contract ให้ sub agent แคบพอและครบพอ
  • รวมผลจาก sub agent ด้วย judgment ของ main agent ไม่ paste ผลลัพธ์ดิบ
  • ไม่ใช้ operations role เว้นแต่ผู้ใช้ระบุ operations intent
  • change มีขอบเขตเล็กและ maintainable
  • review ครอบคลุม correctness, business, security, performance และ testability
  • รัน test หรือ validation ที่เหมาะสม และอธิบาย test ที่ไม่ได้รัน
  • อัปเดต memory เฉพาะ durable knowledge ที่ควรจำข้ามงาน
  • final answer รวมผลจาก sub agent เป็นคำตอบเดียวที่ actionable
